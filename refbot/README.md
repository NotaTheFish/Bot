# refbot — реферальный маркетинг-бот

> **Что где лежит:**
> `setup.sql` — единственный SQL. Накатывается один раз, идемпотентен, гоняй хоть десять раз.
> `main.py` — точка входа. `config.py` — все цифры экономики и антифрода.
> Остальное трогать не надо.

Стек: aiogram 3, asyncpg, PostgreSQL (префикс `rb_`), Railway.

## Запуск

1. **База:** Railway → Postgres → Console → Upload `setup.sql` → `psql -U postgres -d railway -f /setup.sql`
2. Переменные на Railway: `BOT_TOKEN`, `DATABASE_URL`, `SUPER_ADMINS` (твой tg_id — **без него не увидишь админку**), опционально `PAYOUT_CHAT_ID`
3. **BotFather → Group Privacy → OFF** (иначе бот не увидит `!рулетка`)
4. Бота в чат → админ с правом **«Пригласительные ссылки»**
5. Создатель чата пишет в чате `/bind`
6. `python main.py`

## Экономика (проверено расчётом)

| | Грибы | Коины |
|---|---|---|
| За реферала | 5 000 | 100 000 |
| Мин. вывод | 100 000 | 2 000 000 |
| Рулетка min/мода/max | 50 / 250 / 10 000 | 1k / 5k / 200k |
| **EV прокрутки** | **347** | **6 949** |

Коины = грибы × 20 во всех местах. Одна таблица шансов, один множитель.

Полосы рулетки (плотность падает в обе стороны от 250):

```
    50-120    15.00%   0.21429 %/ед
   120-200    22.00%   0.27500 %/ед
   200-300    34.00%   0.34000 %/ед  ← мода
   300-500    18.00%   0.09000 %/ед
   500-1000    8.00%   0.01600 %/ед
  1000-3000    2.50%   0.00125 %/ед
  3000-10000   0.50%   0.00007 %/ед
```

**Считай перед запуском:** 200 активных × 347 = ~70k грибов/сутки только с рулетки.
Это 0.7 вывода в день из воздуха. Крути `ROULETTE_BANDS` в `config.py`,
`python roulette.py` пересчитает EV.

## Что защищает деньги

| Риск | Защита |
|---|---|
| Двойное начисление | `rb_ledger.idempotency_key UNIQUE`. Повтор проводки = no-op |
| Баланс в минус | `CHECK (amount >= 0)` на `rb_balances` |
| Двойная прокрутка (спам командой) | `UNIQUE (tg_id, spin_day)` — гонку ловит БД, не код |
| Двойной вывод | `UNIQUE INDEX ... WHERE status='pending'` |
| Админ жмёт кнопку со старой суммой | `version` в `callback_data`, растёт при каждом изменении |
| Два админа подтвердили одновременно | `SELECT ... FOR UPDATE` + проверка статуса |
| Ссылку перепродали | `member_limit=1` + сверка `visitor_id` с тем, кто реально зашёл |
| Перепривязка реферала к другому | `rb_targets.first_inviter_id` — вечная, первый навсегда |
| Фарм на возвратах | `burned` после «выплатили → ушёл». Навсегда, для всех |
| Мультиферма | `BURST_COUNT` за `BURST_WINDOW_MIN` → `flagged` → ручная проверка |
| Слив бюджета за ночь | `daily_budget_mush` — превышен, начисления ждут завтра (не сгорают) |
| Пропущенный `chat_member` | перед выплатой `get_chat_member` — повторная проверка через API |
| Новорегистрации | `MIN_ACCOUNT_ID_THRESHOLD` (эвристика по tg_id, по умолчанию выкл) |
| Разбор конфликтов | `rb_ledger` + `rb_audit` — append-only, ничего не удаляется |

**Списание происходит только в момент, когда админ жмёт «Подтвердить».**
До этого юзер меняет сумму сколько хочет — старая карточка удаляется, новая приходит.

## Про премиум-эмодзи

Бот может слать `custom_emoji` **только если у него куплен доп. юзернейм на Fragment**.
Это ограничение Bot API, обойти нельзя. Заполнишь `PREMIUM_EMOJI` в `config.py` —
подхватится; пусто — автоматом обычные эмодзи, ничего не сломается.

## Решения, которые я принял за тебя

1. **Валюта фиксируется в момент входа реферала**, не выплаты. Переключил тумблер —
   висящие холды остались в старой валюте. Иначе нужен пересчёт, а пересчёт = дыра.
2. **Первый пригласивший — навсегда.** Реферал ушёл, вернулся по чужой ссылке —
   деньги всё равно первому. Без этого мультиаккаунты гоняют одного «реферала» по кругу.
3. **Выплату при burn не отбираем.** Ты не говорил отбирать; забирать уже начисленное —
   гарантированный скандал в чате.
4. **Один активный вывод на юзера.** Иначе двойное резервирование одних и тех же грибов.

## Кастомизация (без деплоя)

«🛠 Админка → 🎨 Кастомизация». Всё живёт в `rb_settings`, применяется сразу.

- **😀 Эмодзи** — 12 слотов (грибы, коины, баланс, холд, рефералы, рулетка…).
  Отправляешь эмодзи сообщением — бот сохраняет. Отправил **премиум-эмодзи** —
  бот сам вытащит `custom_emoji_id` из entities и запомнит его вместе с символом-фолбэком.
- **🏷 Названия валют** — «Грибы»/«Коины» меняются на что угодно.
- **📝 Шаблон профиля** — свой текст с плейсхолдерами `{e_*}`, `{l_*}`, `{bal_m}`, `{paid}`…
  Валидируется перед сохранением; если всё же сломается — профиль откатится на дефолт,
  юзер ошибки не увидит.
- **🧪 Тест премиум-эмодзи** — сразу видно, принимает их Telegram или нет.
- **♻️ Сброс** — только внешний вид, балансы не трогает.

### Как это работает технически

Telegram **не разрешает `parse_mode` и `entities` одновременно**. Как только нужен
`custom_emoji` (а это entity), HTML-разметка отваливается и `<b>` лезет в текст как есть.
Поэтому `services/render.py` парсит HTML сам и отдаёт один список entities — и жирный,
и премиум вместе. Offsets в UTF-16, эмодзи вне BMP считаются за 2 единицы (на этом
обычно всё и ломается — покрыто тестами, включая ZWJ-эмодзи).

**Ограничение Telegram:** бот шлёт премиум-эмодзи, только если у него куплен
доп. юзернейм на Fragment. Обойти нельзя. Нет юзернейма → Telegram отклоняет entity →
`render.py` ловит ошибку и молча шлёт обычным HTML с символом-фолбэком.
Ничего не ломается, просто не анимировано. Проверяется кнопкой «🧪 Тест».

## Отключение чатов

`/unbind` в чате (владелец) или «🛠 Админка → 📢 Чаты» → карточка → «⚪️ Отключить».

`active=FALSE`, и всё. **Ничего не удаляется:** балансы, леджер, рефералы, холды,
реф-ссылки, `burned`-метки. Отключённый чат не выдаёт ссылок, не начисляет и не крутит
рулетку. Включил обратно — продолжается с того же места. Пишется, кто и когда отключил.

## Команды

**Юзер (ЛС):** `/start` — меню, профиль, ссылка, тумблер валюты, рефералы, вывод
**Чат:** `!рулетка` / `!шимм рулетка` / `!шим рулетка`, `!баланс`
**Админ:** `/bind`, `/unbind` (в чате, только создатель), `/addadmin` (реплаем),
`/rban <id|реплай> причина`, `/runban`, `/approve_<id>`, `/deny_<id>`, кнопка «🛠 Админка»
(топ-25, поиск юзера с историей операций, очередь проверки, сводка обязательств,
управление чатами, кастомизация)

**Админку видят:** все из `SUPER_ADMINS` + все, кто есть в `rb_admins` (попадают туда
через `/bind` и `/addadmin`). Не видишь кнопку — проверь, что твой tg_id в `SUPER_ADMINS`.

## Несколько чатов

Работает из коробки. Всё завязано на пару `(chat_id, user_id)`, а не на юзера:

- Своя реф-ссылка на каждый чат (`rb_ref_links` — `UNIQUE (chat_id, inviter_id)`)
- Один и тот же друг приносит награду **в каждом чате отдельно**
- `burned` действует только в рамках своего чата
- В боте — выбор чата кнопками, в профиле разбивка по чатам

**Касса общая на все чаты.** Заработал в Чате 1 и Чате 2 → один баланс → один вывод.
Подходит, пока **все чаты твои**. Если появится чат с другим владельцем — он начнёт
платить за чужих рефералов. Тогда балансы надо делить по чатам (`rb_balances` +
`chat_id` в PK). См. комментарий в `handlers/user.py::_payout_chat`.

**Рулетка — раз в сутки на юзера глобально.** Два чата ≠ две прокрутки
(`UNIQUE (tg_id, spin_day)`). Бюджет списывается с того чата, где крутанули;
упёрлись в лимит — прокрутка не тратится, откат транзакции.

`PAYOUT_CHAT_ID` в env — куда слать заявки на вывод. Не задан → самый ранний активный.
