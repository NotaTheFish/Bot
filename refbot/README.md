# refbot — реферальный маркетинг-бот

> **Что где лежит:**
> `setup.sql` — единственный SQL. Накатывается один раз, идемпотентен, гоняй хоть десять раз.
> `main.py` — точка входа. `config.py` — все цифры экономики и антифрода.
> `services/ui.py` — ЕДИНАЯ точка отправки и создания кнопок. В хендлерах не зовём
> `edit_text`/`answer`/`kb.button` напрямую, только `ui.*` и `ui.btn` — иначе
> премиум-эмодзи в этом месте молча пропадут.
> Остальное трогать не надо.

Стек: aiogram 3, asyncpg, PostgreSQL (префикс `rb_`), Railway.

## Запуск

1. **База:** Railway → Postgres → Console → Upload `setup.sql` → `psql -U postgres -d railway -f /setup.sql`
2. Переменные на Railway: `BOT_TOKEN`, `DATABASE_URL`, `SUPER_ADMINS` (твой tg_id — **без него не увидишь админку**), опционально `PAYOUT_CHAT_ID`, `UNLIMITED_SPIN_IDS`
3. **BotFather → Group Privacy → OFF** (иначе бот не увидит `!рулетка`)
4. Бота в чат → админ с правом **«Пригласительные ссылки»**
5. Создатель чата пишет в чате `/шайнуть`
6. `python main.py`

## Экономика (проверено расчётом)

| | Грибы | Коины |
|---|---|---|
| За реферала | 5 000 | 100 000 |
| Мин. вывод | 100 000 | 2 000 000 |
| Рулетка min/мода/max | 50 / 250 / 10 000 | 1k / 5k / 200k |
| **EV прокрутки** | **347** | **6 949** |

Коины = грибы × 20 во всех местах. Одна таблица шансов, один множитель.

Полосы рулетки (плотность падает в обе стороны от 250):

```
    50-120    15.00%   0.21429 %/ед
   120-200    22.00%   0.27500 %/ед
   200-300    34.00%   0.34000 %/ед  ← мода
   300-500    18.00%   0.09000 %/ед
   500-1000    8.00%   0.01600 %/ед
  1000-3000    2.50%   0.00125 %/ед
  3000-10000   0.50%   0.00007 %/ед
```

**Считай перед запуском:** 200 активных × 347 = ~70k грибов/сутки только с рулетки.
Это 0.7 вывода в день из воздуха. Крути `ROULETTE_BANDS` в `config.py`,
`python roulette.py` пересчитает EV.

## Что защищает деньги

| Риск | Защита |
|---|---|
| Двойное начисление | `rb_ledger.idempotency_key UNIQUE`. Повтор проводки = no-op |
| Баланс в минус | `CHECK (amount >= 0)` на `rb_balances` |
| Двойная прокрутка (спам командой) | `UNIQUE (tg_id, spin_day)` — гонку ловит БД, не код |
| Двойной вывод | `UNIQUE INDEX ... WHERE status='pending'` |
| Админ жмёт кнопку со старой суммой | `version` в `callback_data`, растёт при каждом изменении |
| Два админа подтвердили одновременно | `SELECT ... FOR UPDATE` + проверка статуса |
| Ссылку перепродали | `member_limit=1` + сверка `visitor_id` с тем, кто реально зашёл |
| Перепривязка реферала к другому | `rb_targets.first_inviter_id` — вечная, первый навсегда |
| Фарм на возвратах | `burned` после «выплатили → ушёл». Навсегда, для всех |
| Мультиферма | `BURST_COUNT` за `BURST_WINDOW_MIN` → `flagged` → ручная проверка |
| Слив бюджета за ночь | `daily_budget_mush` — превышен, начисления ждут завтра (не сгорают) |
| Пропущенный `chat_member` | перед выплатой `get_chat_member` — повторная проверка через API |
| Новорегистрации | `MIN_ACCOUNT_ID_THRESHOLD` (эвристика по tg_id, по умолчанию выкл) |
| Разбор конфликтов | `rb_ledger` + `rb_audit` — append-only, ничего не удаляется |

**Списание происходит только в момент, когда админ жмёт «Подтвердить».**
До этого юзер меняет сумму сколько хочет — старая карточка удаляется, новая приходит.

## Про премиум-эмодзи

**Bot API 9.4 (09.02.2026):** бот может слать `custom_emoji`, если **у владельца бота
есть Telegram Premium** — в личку, группы и супергруппы. Fragment-юзернейм — лишь
альтернативный путь, обязательным он больше не является. Каналы не поддерживаются.

Настраивается в боте: «🛠 Админка → 🎨 Кастомизация → 😀 Эмодзи» — отправляешь
премиум-эмодзи сообщением, бот сам вытащит ID. «🧪 Тест» проверяет отправку
и редактирование по отдельности и показывает реальный ответ Telegram.

## Решения, которые я принял за тебя

1. **Валюта фиксируется в момент входа реферала**, не выплаты. Переключил тумблер —
   висящие холды остались в старой валюте. Иначе нужен пересчёт, а пересчёт = дыра.
2. **Первый пригласивший — навсегда.** Реферал ушёл, вернулся по чужой ссылке —
   деньги всё равно первому. Без этого мультиаккаунты гоняют одного «реферала» по кругу.
3. **Выплату при burn не отбираем.** Ты не говорил отбирать; забирать уже начисленное —
   гарантированный скандал в чате.
4. **Один активный вывод на юзера.** Иначе двойное резервирование одних и тех же грибов.

## Кастомизация (без деплоя)

«🛠 Админка → 🎨 Кастомизация». Всё живёт в `rb_settings`, применяется сразу.

- **😀 Эмодзи** — 12 слотов (грибы, коины, баланс, холд, рефералы, рулетка…).
  Отправляешь эмодзи сообщением — бот сохраняет. Отправил **премиум-эмодзи** —
  бот сам вытащит `custom_emoji_id` из entities и запомнит его вместе с символом-фолбэком.
- **🏷 Названия валют** — «Грибы»/«Коины» меняются на что угодно.
- **📝 Шаблон профиля** — свой текст с плейсхолдерами `{e_*}`, `{l_*}`, `{bal_m}`, `{paid}`…
  Валидируется перед сохранением; если всё же сломается — профиль откатится на дефолт,
  юзер ошибки не увидит.
- **🧪 Тест премиум-эмодзи** — сразу видно, принимает их Telegram или нет.
- Премиум в тексте: любой эмодзи станет премиумом, если для него задана свободная
  замена. Панель «♻️ Свободные замены» показывает, какие из реально используемых
  символов ещё без премиума. Настройки применяются **везде**: профиль, рулетка, `!баланс`, меню, заявки на вывод,
  уведомления о рефералах, админка. Хардкода эмодзи в коде не осталось.
- **♻️ Свободные замены — главное.** Мапят ЛЮБОЙ символ на премиум-эмодзи.
  Замапил 📊 один раз — он станет премиумом везде, где встретится: в тексте,
  на кнопке, в админке, в рулетке. Слоты нужны только для смысловых вещей
  («какое эмодзи значит грибы»), всё остальное закрывается заменами.
  Отправляешь одним сообщением: обычное эмодзи, затем премиум.
- **Кнопки с премиумом — автоматически.** Bot API 9.4 добавил `icon_custom_emoji_id`
  для `InlineKeyboardButton` (нужна aiogram ≥ 3.29). `ui.btn()` сам смотрит на первый
  символ текста кнопки: замаплен — ставит иконку и убирает эмодзи из текста (иначе
  был бы дубль); не замаплен — оставляет как есть. Прописывать руками ничего не надо,
  включая кнопки, которых ещё нет.
- **♻️ Сброс** — только внешний вид, балансы не трогает.

### Почему нет ASCII-рамок

Telegram рендерит текст **пропорциональным шрифтом**. `═` и `║` узкие, эмодзи — широкие
и разной ширины на Windows / Android / iOS. Рамку `╔═══╗` с эмодзи внутри выровнять
нельзя в принципе — она всегда будет ползти, и на каждом устройстве по-своему.
Поэтому рамку рисует сам Telegram через `<blockquote>`: вертикальная полоса слева,
выровнена всегда и везде. Плюс `▰▱`-полоса прогресса — одинаковые символы, не разъезжаются.

### Как это работает технически

Telegram **не разрешает `parse_mode` и `entities` одновременно**. Как только нужен
`custom_emoji` (а это entity), HTML-разметка отваливается и `<b>` лезет в текст как есть.
Поэтому `services/render.py` парсит HTML сам и отдаёт один список entities — и жирный,
и премиум вместе. Offsets в UTF-16, эмодзи вне BMP считаются за 2 единицы (на этом
обычно всё и ломается — покрыто тестами, включая ZWJ-эмодзи).

**Права на премиум-эмодзи.** Bot API 9.4 (09.02.2026) разрешил ботам `custom_emoji`,
**если у владельца бота есть Telegram Premium** — в личку, группы и супергруппы.
Раньше требовался доп. юзернейм на Fragment; теперь это лишь альтернативный путь.
Каналы не поддерживаются.

Дока говорит "messages **directly sent** by the bot" и молчит про `editMessageText`.
Анимация рулетки построена на редактировании, поэтому «🧪 Тест» в админке проверяет
**оба пути отдельно** и показывает настоящий ответ Telegram. Если редактирование
премиум не тянет — финальную карточку рулетки надо слать отдельным сообщением.

Если Telegram всё же откажет — `render.py` ловит ошибку и шлёт обычным HTML
с символом-фолбэком. Ничего не ломается, просто не анимировано.

## Рулетка в любом чате

`!шайн` работает **без `/шайнуть` и без прав админа** — достаточно, чтобы бот был в чате
и Group Privacy был выключен. Рефералка при этом по-прежнему требует `/шайнуть` и право
«Пригласительные ссылки».

**Почему это не дыра.** Прокрутка одна в сутки **на юзера глобально**
(`UNIQUE (tg_id, spin_day)`), поэтому сто чатов не дадут сто выплат — расход упирается
в число живых людей, а не чатов. Сверху ещё общий потолок: `FREE_ROULETTE_BUDGET`
(по умолчанию 300k грибов) на **все** непривязанные чаты вместе за сутки.
`FREE_ROULETTE=0` выключает их совсем.

Счётчики свободных чатов живут в `rb_free_budget` и `rb_free_chats` — **специально
отдельно от `rb_chats`**. Если бы случайные чаты писались туда, они полезли бы
в «🔗 Мою ссылку», в список чатов админки и в маршрутизацию выводов.

Свободные чаты видны в «📊 Сводка»: потрачено/потолок и сколько чатов.
`/shine_off` в чате (админ чата) блокирует рулетку именно там.

## Тестовый режим: безлимитная рулетка

`UNLIMITED_SPIN_IDS=111,222` — этим ID лимит «раз в сутки» не действует.

**Это обход защиты от накрутки.** Суточный бюджет чата на них всё равно действует,
так что бесконечно слить нельзя, но перед боевым запуском переменную надо **очистить**.
Бот при старте орёт в лог, если она заполнена.

Реализация не ломает `UNIQUE (tg_id, spin_day)`: для тестовых ID сегодняшняя строка
в `rb_spins` удаляется перед вставкой. Индекс остаётся боевым для всех остальных,
а полная история прокруток лежит в `rb_ledger` — там у каждой свой `spin:<id>`.

## Отключение чатов

`/отшайнуть` в чате (владелец) или «🛠 Админка → 📢 Чаты» → карточка → «⚪️ Отключить».

`active=FALSE`, и всё. **Ничего не удаляется:** балансы, леджер, рефералы, холды,
реф-ссылки, `burned`-метки. Отключённый чат не выдаёт ссылок, не начисляет и не крутит
рулетку. Включил обратно — продолжается с того же места. Пишется, кто и когда отключил.

## Реферальные ссылки: только для своих

«🔗 Моя ссылка» показывает кнопки **всех зарегистрированных чатов**. Нажал —
бот проверяет через `get_chat_member`, состоишь ли ты в этом чате:

- состоишь → выдаёт реферальную ссылку именно на этот чат;
- не состоишь → даёт ссылку на вступление и говорит «сначала вступи».

Ссылка на вступление **не реферальная** и не пишется в `rb_invites` — человек пришёл
сам, награду за него никто не получит. `on_join` не найдёт её по имени и пройдёт мимо.

Так случайно нашедший бота не раздаёт ссылки на чаты, о которых ничего не знает.

## Еженедельный конкурс по активности

`/шимшайнуть` в чате (админ чата) — подключить. `/отшимшайнуть` — отключить.
**Это НЕ рефералка:** отдельная таблица `rb_contest_chats`, чат не появится
ни в «🔗 Моей ссылке», ни в маршрутизации выводов. Хочешь и рефералку — отдельно
`/шайнуть`.

- Период: пн 00:00 → вс 23:59 в зоне `CONTEST_TZ` (по умолчанию `Europe/Moscow`)
- От **500** сообщений — участие. 1 билет за каждые **50** (500 сообщений = 10 билетов)
- `/неделя` — таблица активности, билеты, твоё место
- В конце периода бот сам постит итог, снимает **свой** прошлый закреп (чужие не трогает),
  закрепляет новый и вешает кнопку «Запустить рулетку» — жмёт админ
- Цикл вечный: период закрывается **по времени**, а не по наличию сообщений.
  Тихая неделя тоже получит анонс («никто не набрал»). Останавливает только
  `/отшимшайнуть`
- Победитель взвешенно-случайный: больше билетов — выше шанс, но не гарантия
- Приз: **100 000 грибов** или **2 000 000 коинов** (по валюте победителя) — курс 20×, как везде
- После розыгрыша — кнопка со ссылкой на победителя

### Когда стартует первый конкурс

Бот не «запускается в понедельник» — он всегда знает по календарю, к какому периоду
относится момент. Вопрос только в первой неделе:

- Привязал `/шимшайнуть` **в понедельник 00:00** — первая же неделя полная.
- Привязал **в среду** — текущая неделя неполная (пн-вт бот не видел). По умолчанию
  (`CONTEST_COUNT_PARTIAL_FIRST=0`) она **пропускается**, первый конкурс — по итогам
  следующей полной недели. Так счёт честный. `/шимшайнуть` прямо пишет дату первого
  розыгрыша.
- Нужен розыгрыш уже по неполной неделе — `CONTEST_COUNT_PARTIAL_FIRST=1`.

Часовой пояс — `CONTEST_TZ`. Стоит `Europe/Moscow`: «понедельник 00:00» считается
по Москве. Поменяешь зону — сдвинется граница недели.

### Почему ничего не обнуляется

«Статистика обнуляется в 00:00, но билеты живут до нажатия кнопки» — неразрешимо,
если хранить «текущую неделю» и чистить её. Поэтому счётчики пишутся с ключом
`period_start`: новая неделя — просто новые строки, старый период лежит нетронутым
хоть месяц. Нажмёшь кнопку через две недели — разыграется корректно.

`UNIQUE (chat_id, period_start)` на `rb_week_draws` — второго розыгрыша того же
периода не будет физически. На нём же держится и «ровно один анонс за период»:
воркер просыпается раз в минуту и просто пытается создать розыгрыш, а отсекает
дубли база, а не проверка в коде. Приз идёт через `db.apply` с ключом `draw:<id>` —
двойная выплата невозможна.

### Тестовый режим

`CONTEST_TEST_MINUTES=1` — «неделя» длится 1 минуту. Меняется **только** функция
границ периода, вся остальная логика та же, поэтому тестируешь боевой код.
Очистить перед боевым запуском — бот орёт в лог, если переменная стоит.

Для теста ставь **обе** переменные:

```
CONTEST_TEST_MINUTES=1
CONTEST_MIN_MSGS=5
CONTEST_MSGS_PER_TICKET=1
```

`CONTEST_MSGS_PER_TICKET=1` важен: с боевыми 50 за билет и порогом 5 у тебя будет
ровно 1 билет у каждого — шансы поровну, разницу в весах не увидишь.
Бот предупредит в логе, если порог окажется ниже цены билета.

**Инвариант:** прошёл порог → билетов всегда минимум 1. Состояния «участник с нулём
билетов» не существует ни при каких настройках (иначе конкурс молча не состоится).

Своего антиспама у конкурса нет намеренно: за флудом в чате следит отдельный бот.

## Команды

**Юзер (ЛС):** `/start` — меню, профиль, ссылка, тумблер валюты, рефералы, вывод
**Чат:** `!шайн` / `!шимм шайн` / `!шим шайн`, `!баланс` — работают в **любом** чате,
без `/шайнуть` и без админки боту (нужен только Group Privacy = OFF)
**Конкурс:** `/шимшайнуть`, `/отшимшайнуть` (админ чата; `/cbind`, `/cunbind` — алиасы), `/неделя`

### Баны

«🛠 Админка → 🚫 Баны» — список забаненных (имя + ID + причина). Внутри:
- «🔗 Ссылки на профили» — клавиатура меняется на кнопки-ссылки `tg://user` по каждому,
  «⬅️ Назад к списку» возвращает обычный вид
- «♻️ Разбанить по ID» — ввод ID (он написан рядом с именем)

При поиске юзера («🔍 Найти юзера») под карточкой — тумблер: **🥳 бан** банит,
после чего становится **😡 бан** и разбанит. Бот уведомляет забаненного/разбаненного.
Бан гасит незавершённые холды.

### Промокод (тест вывода)

`PROMO_CODE=СЛОВО` на Railway — активируется командой `/promo СЛОВО` в боте.
Даёт **100 000 грибов** или **2 000 000 коинов** (по валюте юзера). Безлимитный по
числу людей, но один человек активирует один раз (`rb_promo_used`). Удалил переменную —
промокод перестал работать. Нужен, чтобы начислить себе баланс и проверить вывод.

### Заявки на вывод (чтобы не потерять)

«🛠 Админка → 💸 Заявки на вывод» — список всех открытых заявок с суммой и сколько
ждут. Если админ проспал и карточки затерялись в ЛС, тут они все; кнопка на каждой
пришлёт карточку с «Подтвердить/Отклонить» заново. Payout-админам тоже доступно.

Карточка заявки теперь с кнопкой-ссылкой на профиль игрока (имя = ссылка на `tg://`
или `t.me/username`).

### Устойчивость к флуд-контролю

Telegram придерживает ботов за частые сообщения (`TelegramRetryAfter`). Рулетка шлёт
4 сообщения за прокрутку, в оживлённом чате это быстро упирается в лимит. Раньше это
исключение роняло обработчик — бот «зависал». Теперь:
- `render._safe` ждёт указанное Telegram время и повторяет;
- глобальный `@dp.errors()` ловит всё, что всё же вылетело, и бот продолжает жить;
- баланс в рулетке начисляется в транзакции ДО анимации — выигрыш не теряется,
  даже если кадры не отправились.

### Payout-админ

`PAYOUT_ADMINS=111,222` на Railway — эти люди получают карточки заявок на вывод в ЛС
и могут их подтверждать/отклонять. **Больше ничего:** ни банов, ни статистики, ни
настроек — в саму админку они не попадают. Карточка вывода теперь уходит всем сразу
(владельцу чата + payout-админам); кто первый нажал, у того и проведётся, у остальных
кнопка станет неактивной.

**Админ:** `/шайнуть`, `/отшайнуть` (в чате, только создатель; `/bind` и `/unbind` — алиасы), `/addadmin` (реплаем),
`/shine_off`, `/shine_on` (выключить/включить рулетку в конкретном чате),
`/rban <id|реплай> причина`, `/runban`, `/approve_<id>`, `/deny_<id>`, кнопка «🛠 Админка»
(топ-25, поиск юзера с историей операций, очередь проверки, сводка обязательств,
управление чатами, кастомизация)

**Админку видят:** все из `SUPER_ADMINS` + все, кто есть в `rb_admins` (попадают туда
через `/bind` и `/addadmin`). Не видишь кнопку — проверь, что твой tg_id в `SUPER_ADMINS`.

## Несколько чатов

Работает из коробки. Всё завязано на пару `(chat_id, user_id)`, а не на юзера:

- Своя реф-ссылка на каждый чат (`rb_ref_links` — `UNIQUE (chat_id, inviter_id)`)
- Один и тот же друг приносит награду **в каждом чате отдельно**
- `burned` действует только в рамках своего чата
- В боте — выбор чата кнопками, в профиле разбивка по чатам

**Касса общая на все чаты.** Заработал в Чате 1 и Чате 2 → один баланс → один вывод.
Подходит, пока **все чаты твои**. Если появится чат с другим владельцем — он начнёт
платить за чужих рефералов. Тогда балансы надо делить по чатам (`rb_balances` +
`chat_id` в PK). См. комментарий в `handlers/user.py::_payout_chat`.

**Рулетка — раз в сутки на юзера глобально.** Два чата ≠ две прокрутки
(`UNIQUE (tg_id, spin_day)`). Бюджет списывается с того чата, где крутанули;
упёрлись в лимит — прокрутка не тратится, откат транзакции.

`PAYOUT_CHAT_ID` в env — куда слать заявки на вывод. Не задан → самый ранний активный.

- Кастомизация шаблонов рулетки и карточки ссылки (пока настраивается только профиль)
- Экспорт леджера в CSV
- Anti-flood middleware (пока только БД-гарантии)
